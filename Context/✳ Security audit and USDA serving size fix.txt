  ---
  How the APIs are safe now

  The whole design is "the browser never holds a usable secret, and only you can spend the keys." Here's the layered
  picture for a request like voice or food search:

  Your browser  →  Supabase Edge Function (proxy)  →  Anthropic / USDA
  (no API key)     (adds the real key server-side)     (sees only the proxy)

  1. The keys aren't in the browser at all.
  Before, the Anthropic/USDA keys were compiled into the JavaScript bundle (VITE_… vars) — anyone could open
  devtools and copy them. Now they live only as Supabase secrets (ANTHROPIC_API_KEY, USDA_API_KEY), encrypted
  server-side. The function attaches the key inside the request, after it leaves your browser. I verified the live
  bundle: it contains only the public Supabase URL/anon key — zero API keys.

  2. The only thing the frontend has is meant to be public.
  VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY are designed to be exposed. They can't read or write data on their
  own — Row-Level Security on every table restricts rows to their owner, so the anon key alone is useless to an
  attacker.

  3. Every proxy call must carry a valid login (→ 401).
  supabase-js automatically attaches your session JWT to each function call. The function calls auth.getUser(token)
  and rejects anything without a valid, unexpired token with 401 — before touching the paid key.

  4. Even a valid login must be you (→ 403).
  On top of the JWT check, the function compares the authenticated email against ALLOWED_EMAIL
  (brandonmoran603@gmail.com). Anyone else — even if they sign up for their own account — gets 403. So the keys can
  only be spent by you.

  5. Requests must come from your site (CORS).
  The functions only accept browser requests from https://slomhub.vercel.app (and localhost:5173 for dev). A
  malicious page on another domain can't call your proxy from a victim's browser.

  6. Defense in depth.
  verify_jwt = true is also pinned at the Supabase gateway (a second JWT check before the function runs), and the
  new password-only login removes the magic-link path as a way to obtain a session.

  What an attacker can and can't do now

  ┌───────────────────────────────────────────┬──────────────────────────┐
  │                  Attempt                  │          Result          │
  ├───────────────────────────────────────────┼──────────────────────────┤
  │ Read the keys from the shipped JS         │ ❌ not there             │
  ├───────────────────────────────────────────┼──────────────────────────┤
  │ Call your proxy with no login             │ ❌ 401                   │
  ├───────────────────────────────────────────┼──────────────────────────┤
  │ Sign up their own account and call it     │ ❌ 403 (not whitelisted) │
  ├───────────────────────────────────────────┼──────────────────────────┤
  │ Call the proxy from another website       │ ❌ blocked by CORS       │
  ├───────────────────────────────────────────┼──────────────────────────┤
  │ Use the public anon key to read your data │ ❌ blocked by RLS        │
  └───────────────────────────────────────────┴──────────────────────────┘
